# Show HN Template

## Title

```
Show HN: {Concise description of what it does} [{key technology if relevant}]
```

Keep under 80 characters. Describe function, not quality.

---

## Top-Level Creator Comment

Post this as the first comment on your submission:

---

**What it is:**
{One sentence. What does the thing do? Be specific. "A command-line tool that generates database migration scripts from schema diffs" not "A powerful developer tool."}

**Why I built it:**
{2-3 sentences. What problem motivated this? What was the gap in existing tools? Personal context helps: "I was managing 40 database schemas at work and got tired of writing migration scripts by hand."}

**How it works:**
{3-5 sentences or a short list. The technical architecture overview. Key technologies, interesting design decisions, how the pieces fit together.}

For example:
- Parses schema files using {parser}
- Generates a diff tree using {algorithm}
- Outputs idiomatic SQL for {databases supported}
- Written in {language}, runs as a single binary with no dependencies

**What's interesting technically:**
{2-3 sentences. What did you learn building this? What problem was harder than expected? What design decision are you most pleased or uncertain about? This is what HN readers engage with most.}

**Current state:**
{Honest assessment. What works well. What doesn't work yet. Known limitations. What's on the roadmap.}

- Works: {feature 1}, {feature 2}, {feature 3}
- In progress: {feature}
- Known issues: {limitation 1}, {limitation 2}
- Not yet: {future feature}

**What I'm looking for:**
{Specific feedback request. "I'd love feedback on the diff algorithm approach" or "If you work with large schemas, I'd appreciate hearing about edge cases I might be missing." Not just "Feedback welcome."}

Links:
- GitHub: {url}
- Demo: {url} (if applicable)
- Docs: {url} (if applicable)

---

## Notes on the Creator Comment

- Length: 200-400 words. Long enough to be substantive, short enough to be read.
- Tone: Direct, technical, honest. See `hn-voice.md`.
- Do not include pricing, calls-to-action, or marketing copy in the comment.
- If there is a paid tier, mention it briefly and factually: "There's a hosted version for teams. The core tool is MIT-licensed."
- Respond to every substantive comment on your post. HN rewards engaged creators.
- If someone finds a bug or limitation, acknowledge it immediately. "Good catch, I'll fix that" goes a long way.
- Do not ask for upvotes, shares, or support. Let the work speak for itself.
